================================================================================
  WHAT'S DONE WELL
================================================================================

• Solid concept — a "Dual-AI" app (Wingman + Consultant) is creative and
  technically ambitious for an FYP. It demonstrates real-time audio processing,
  graph databases, vector search, and LLM integration.

• Clean project structure — separation into screens/, services/, widgets/,
  providers/, and theme/ is a good architectural choice and shows awareness of
  Flutter best practices.

• Material 3 theming — properly uses ColorScheme.fromSeed(), supports light and
  dark mode, and even lets the user pick an accent color that persists via
  SharedPreferences.

• Auth flow with AuthGate — the StreamBuilder approach in main.dart that
  reactively switches between Splash → Login → Home based on Supabase auth
  state is the recommended pattern.

• Profile caching — AuthService caches the profile in SharedPreferences and
  merges updates locally before writing to the server. This reduces network
  calls and provides a snappy offline-first feel.

• Reusable widget library — AppButton, AppInput, AppCard, SocialButton,
  AppLogo, FadeSlideTransition are all well-abstracted and keep screens clean.

• Nice animations — using flutter_animate with staggered intervals on the
  home screen, login screen, and sessions list gives a polished feel.

• Connection health system — ConnectionService periodically pings the backend
  every 30 seconds and reflects status with a color-coded avatar border. This
  is a UX-conscious design.

• About screen design — the SliverAppBar, glassmorphism cards, and ambient
  background blobs show real effort in UI design.

• Server architecture — services are cleanly separated (GraphService,
  VectorService, BrainService, SessionService) which is good for a backend.


================================================================================
  CRITICAL ISSUES (Fix ASAP)
================================================================================

• HARDCODED API KEYS IN SOURCE CODE
  - main.dart lines 26-27: Supabase URL and anon key are hardcoded as constants.
  - deepgram_service.dart line 11: Deepgram API key is hardcoded as a static
    const string directly in the Dart source.
  - secrets.txt contains ALL API keys in plaintext (Deepgram, LiveKit, Groq,
    Ngrok, Supabase). Even though it's in .gitignore, the Supabase key in
    main.dart is NOT gitignored.
  - FIX: Use flutter_dotenv or --dart-define flags for the Flutter app.
    Use python-dotenv or environment variables for the server. Never commit
    secrets to version control. If these keys have been pushed to Git at any
    point, they should be rotated immediately.

• CORS WIDE OPEN ON BACKEND
  - server.py line 391: allow_origins=["*"], allow_methods=["*"],
    allow_headers=["*"]. This allows any origin to call your API.
  - FIX: Restrict CORS to your frontend's domain/IP.

• NO INPUT VALIDATION / RATE LIMITING ON SERVER
  - The /ask_consultant and /process_transcript_wingman endpoints accept any
    user_id and any length of text with no validation, no rate limiting.
  - A malicious user could spam the Groq API, rapidly consuming your quota.
  - FIX: Add input length limits, rate limiting (e.g. slowapi), and validate
    that the user_id corresponds to an authenticated Supabase user.

• NO AUTHENTICATION ON SERVER ENDPOINTS
  - The server has no auth middleware. Anyone who discovers the ngrok URL can
    call /ask_consultant, /process_transcript_wingman, /getToken freely.
  - FIX: Pass the Supabase JWT token in the Authorization header and verify it
    server-side before processing requests.

  ARCHITECTURE & CODE QUALITY IMPROVEMENTS

• DUPLICATE CHAT BUBBLE CODE
  - The same chat bubble widget (with alignment, border radius, box shadow,
    color logic) is copy-pasted across:
      - new_session_screen.dart (lines 344-383)
      - consultant_screen.dart (lines 239-269)
      - sessions_screen.dart (lines 338-368)
  - FIX: Extract a reusable ChatBubble widget into widgets/chat_bubble.dart.

• INCONSISTENT NAVIGATION APPROACH
  - Some screens use named routes (Navigator.pushNamed) while others use
    MaterialPageRoute (Navigator.push with builder). The settings screen does
    both (line 92 vs. line 149).
  - FIX: Pick one approach and be consistent. Consider using go_router for
    type-safe, declarative routing.

• EXCESSIVE print() STATEMENTS
  - There are 30+ print() calls scattered across the codebase (services,
    server.py, screens). These are fine for development but should not go into
    a production/demo build.
  - FIX: Replace with a proper logging framework.
    Flutter: use the dart:developer log() function or a package like logger.
    Python: use the standard logging module instead of print().

• AuthService IS SINGLETON BUT OTHER SERVICES USE PROVIDER
  - AuthService.instance is a manually-implemented singleton, while
    ConnectionService, ApiService, LiveKitService, DeepgramService all use
    Provider. This inconsistency makes dependency injection uneven.
  - FIX: Either make AuthService a Provider too, or document why it's different.

• ConnectionService PINGS ROOT "/" BUT LOG SAYS "/health"
  - connection_service.dart line 84 prints "Pinging $_serverUrl/health ..."
    but line 86 actually pings "$_serverUrl/" (just root).
  - FIX: Either implement a /health endpoint on the server or update the log.

• DEPRECATED API USAGE
  - consultant_screen.dart line 47 uses WidgetsBinding.instance.window which
    is deprecated in newer Flutter versions.
  - FIX: Use View.of(context) or MediaQuery instead.

• NO ERROR BOUNDARY / GLOBAL ERROR HANDLING
  - If any screen throws an unhandled exception, the app will show a red error
    screen. There's no FlutterError.onError or runZonedGuarded wrapper.
  - FIX: Add a global error handler in main() and a fallback error widget.

• PROVIDER REBUILD ON UPDATE (LiveKitService)
  - main.dart lines 57-60: ChangeNotifierProxyProvider creates a NEW
    LiveKitService instance on every update. This means every time ApiService
    changes, the LiveKit connection could be lost.
  - FIX: Reuse the previous instance: update should return
    previous!..updateApiService(api) rather than creating a new instance.

• CLIENT-SIDE SUPABASE QUERIES IN SCREENS
  - consultant_screen.dart (lines 77-83) and sessions_screen.dart (lines 80-85,
    166-171) directly call Supabase.instance.client from within screen widgets.
  - FIX: Move all database queries into services (e.g. SessionService) to keep
    screens as pure UI. This also makes testing easier.

• NO requirements.txt / pyproject.toml FOR SERVER
  - server/ has server.py but no requirements.txt or dependency file.
  - The README says "pip install list-of-dependencies..." which is vague.
  - FIX: Create a requirements.txt or pyproject.toml with pinned versions.

• MULTIPLE Supabase CLIENT INSTANCES IN SERVER
  - GraphService, VectorService, and SessionService each create their own
    Supabase client instance. This is wasteful.
  - FIX: Create a single shared Supabase client and inject it into services.

================================================================================
  BACKEND / SERVER IMPROVEMENTS
================================================================================

• KNOWLEDGE GRAPH LOADED INTO MEMORY ON EVERY REQUEST
  - server.py: load_graph() is called on every /ask_consultant and
    /process_transcript_wingman request, hitting the DB each time.
  - FIX: Cache graphs in memory with a TTL, or use Redis.

• BLOCKING OPERATIONS IN ASYNC ENDPOINTS
  - brain_svc.get_wingman_advice() and brain_svc.extract_knowledge() are
    synchronous Groq API calls inside async FastAPI endpoints.
  - FIX: Use asyncio.to_thread() to run them in a thread pool, or use the
    async Groq client (groq.AsyncGroq).

• NO RETRY LOGIC FOR API CALLS
  - If the Groq API times out or returns a 429 (rate limit), the server just
    catches the exception and returns a fallback.
  - FIX: Add retry logic with exponential backoff using tenacity.

• GLOBAL MUTABLE STATE (LIVE_SESSIONS dict)
  - server.py line 85: LIVE_SESSIONS is a plain dict. In a multi-worker setup,
    this would break.
  - FIX: For production, use Redis or a database-backed session store.

• nest_asyncio IN PRODUCTION CODE
  - server.py line 39: nest_asyncio.apply() is a hack for Jupyter/Colab
    environments. It should not be in production code.
  - FIX: Remove it or make it conditional on running in a notebook.

• NO SERVER HEALTH MONITORING
  - No /health endpoint, no readiness/liveness probes, no structured logging.
  - FIX: Add a /health endpoint that checks DB connectivity and returns status.

• SENTENCE TRANSFORMER MODEL LOADED ON STARTUP
  - server.py line 184: SentenceTransformer('all-MiniLM-L6-v2') is loaded on
    every server start, which takes time and memory.
  - FIX: Consider lazy loading, or better yet, use Supabase's built-in
    embedding functions if available.
