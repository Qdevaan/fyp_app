 1. CORE CONCEPT
════════════════════════════════════════════════════════════════════════════════
Bubbles is a dual-engine AI assistant app:

  THE WINGMAN (Real-Time Mode)
  ─────────────────────────────
  • Llama 3.1 8B via Groq for sub-100ms inference
  • LiveKit for real-time audio streaming
  • Deepgram Nova-2 for transcription with diarization
  • Porcupine for offline wake-word detection ("Hey Bubbles")
  • Goal: Live strategic advice during conversations (dates, negotiations, etc.)

  THE CONSULTANT (Deep Dive Mode)
  ─────────────────────────────────
  • Llama 3.3 70B via Groq for detailed reasoning
  • RAG architecture: Vector Memory (MiniLM-L6-v2, 384-dim via pgvector)
                    + Knowledge Graph (NetworkX serialized as JSONB)
  • SSE token streaming to Flutter
  • Goal: Deep Q&A with full personal context

  TECH STACK
  ──────────
  Frontend:   Flutter + Provider + Supabase SDK + Material 3
  Backend:    Python FastAPI + Groq + sentence-transformers + NetworkX
  Database:   Supabase (PostgreSQL + pgvector + pg_trgm + Realtime)
  Audio:      LiveKit + Deepgram + Porcupine + SpeechBrain
  Comms:      HTTP REST + Server-Sent Events (SSE) + WebSockets

─── 3A. ARCHITECTURE & STATE MANAGEMENT ───────────────────────────────────────

  PATTERN: Provider-based dependency injection via MultiProvider in main.dart.

  WHAT'S GOOD:
  • Clean provider hierarchy: ConnectionService → ApiService → LiveKitService.
  • ProxyProvider used correctly so LiveKitService gets updated ApiService refs
    without re-creation (preserves connection state).
  • ChangeNotifierProvider for standalone services (Theme, Deepgram, WakeWord).
  • ChangeNotifierProxyProvider2 for VoiceAssistantService with dual deps.
  • Local setState() used only for UI-only state (form fields, toggles).
  • AuthService is a proper singleton (AuthService.instance).

  WHAT'S BAD:
  • ConsultantScreen manages streaming state, chat messages, voice mode, and 
    drawer state ALL locally — should be a dedicated ChatProvider or
    ConsultantProvider to decouple UI from business logic.
  • NewSessionScreen manages Deepgram connection + Realtime subscription + 
    session state locally — same problem, needs a SessionProvider.
  • HomeScreen does Supabase queries inline (_loadInsights) — data fetching 
    logic belongs in a provider or repository layer.
  • No repository pattern: screens talk directly to Supabase client and 
    ApiService — adding a Repository layer would improve testability.

  VERDICT: Provider usage is competent and consistent. The gap is that complex 
  screens have too much business logic embedded, which Provider alone can't fix.
  A StateNotifier/Riverpod migration isn't necessary — just extract screen-level 
  state into dedicated ChangeNotifiers.

─── 3B. NAVIGATION ────────────────────────────────────────────────────────────

  PATTERN: Named routes in MaterialApp.routes + onGenerateRoute for custom 
  transitions (settings slides from left, entities from right).

  ROUTES DEFINED:
  /login, /signup, /verify-email, /profile-completion, /home, /connections,
  /new-session, /consultant, /sessions, /about, /settings, /entities

  WHAT'S GOOD:
  • Custom SlideTransition for settings (←) and entities (→) adds polish.
  • Gesture-based nav on HomeScreen (swipe right → settings, left → entities).
  • AuthGate widget exists for auth-state switching (Login ↔ Home).
  • SplashScreen routes to correct screen based on auth/profile state.

  WHAT'S BAD:
  • No route guards: Any screen can be accessed by name even if user isn't
    authenticated. Someone calling Navigator.pushNamed('/consultant') from
    a login screen wouldn't be blocked.
  • AuthGate exists but isn't used — SplashScreen handles routing manually. 
    The AuthGate class is dead code.
  • No deep link handling: The app registers a deep link scheme 
    (io.supabase.bubbles://login-callback/) for OAuth but doesn't handle 
    other deep links (e.g., opening a specific session from a notification).
  • Route names are hardcoded strings scattered across screens. Should be 
    centralized constants (e.g., AppRoutes.home, AppRoutes.consultant).
  • Swipe gesture threshold (velocity 300) is hardcoded and could conflict 
    with drawer gestures on some screens.

  WHAT COULD BE IMPROVED:
  • Use GoRouter or auto_route for type-safe routing, guards, and deep links.
  • Centralize route definitions in a single file.
  • Add middleware/guards: redirect to /login if no session, redirect to 
    /profile-completion if profile incomplete.
  • Consider shell routes for persistent bottom nav or sidebar on tablet.

─── 3E. SERVICES LAYER ────────────────────────────────────────────────────────

  AuthService (lib/services/auth_service.dart)
  ─────────────
  + Proper singleton pattern.
  + Profile caching with SharedPreferences (offline-friendly).
  + Cache invalidation on sign-out.
  + Merge strategy for upsertProfile (doesn't wipe existing fields).
  - No token refresh logic (relies on Supabase SDK auto-refresh).
  - uploadAvatar has no file size validation.
  - _handleAuthError catches StorageException but the fallback is generic.

  ApiService (lib/services/api_service.dart)
  ──────────
  + Every endpoint has proper timeout.
  + SSE streaming implemented correctly with buffer parsing.
  + Session lifecycle methods (create, process, end) are complete.
  + "ngrok-skip-browser-warning" header on all requests (needed for free ngrok).
  - No retry logic on any endpoint.
  - No request/response interceptor for logging.
  - getLiveKitToken passes userId as query param (should be POST body or header).
  - processAudioChunk has a very tight 5-second timeout.
  - No cancellation tokens — can't abort in-flight requests.

  ConnectionService (lib/services/connection_service.dart)
  ─────────────────
  + Smart URL normalization (adds protocol, appends port for raw IPs).
  + Periodic health checks every 30 seconds.
  + Offline detection via connectivity_plus.
  + Debounced checking (_isChecking guard prevents overlaps).
  - Health check only hits "/" — doesn't verify that DB/Groq are alive.
  - No exponential backoff on connection failures.
  - 30-second polling could be 60s to save battery.

  LiveKitService (lib/services/livekit_service.dart)
  ──────────────
  + Preserves connection across Provider rebuilds (updateApiService pattern).
  + Proper Room/listener lifecycle management.
  + Data event parsing for transcript and assistant response types.
  - No reconnection logic if room disconnects.
  - No handling for remote participant events.
  - connect() silently fails if token fetch returns null.

  DeepgramService (lib/services/deepgram_service.dart)
  ───────────────
  + Direct WebSocket to Deepgram with auth header.
  + Proper audio streaming from device microphone.
  + Diarization enabled (speaker ID extraction from word-level data).
  - API key loaded from .env but HARDCODED in .env file — dangerous if 
    .env is committed (and it is, since .env is listed in flutter assets).
  - Audio encoder set to AAC LC — Deepgram recommends Linear16 for best 
    transcription accuracy. AAC introduces lossy compression artifacts.
  - Speaker mapping is oversimplified: speaker 0 = "user", all others = 
    "other". Doesn't leverage voice enrollment embeddings at all.
  - WebSocket error handler just calls disconnect() — no reconnection.
  - No buffering or silence detection — sends every audio frame immediately.

  VoiceAssistantService (lib/services/voice_assistant_service.dart)
  ─────────────────────
  + Full pipeline: wake word → STT → command parsing → LLM → TTS response.
  + Porcupine lifecycle coordination (stop during STT, restart after TTS).
  + Deepgram Aura TTS integration (3 voice modes: male, female, neutral).
  + Preferences persistence (voice mode, wake word toggle).
  + Navigation intent resolution (Wingman commands → /new-session route).
  + Activate/deactivate lifecycle tied to authentication state.
  - Command matching is fragile keyword detection (contains("wingman"), 
    contains("tell me")) — will false-positive on normal conversation.
  - 300ms hardcoded delay between Porcupine stop and STT start — might 
    not be enough on all devices, might be too much on fast ones.
  - No fallback if Deepgram TTS fails — user gets silence.
  - Pending navigation via string _pendingNavigationRoute is fragile.

  WakeWordService (lib/services/wake_word_service.dart)
  ───────────────
  + Proper lifecycle management with WidgetsBindingObserver (pause/resume).
  + Asset extraction from Flutter bundle to temp directory.
  + Sensitivity configurable (0.7 — good default).
  + Guard against concurrent init calls (_isInitializing flag).
  + Proper cleanup on dispose (delete Porcupine manager).
  - No retry if init fails (Picovoice key expired, etc.).
  - Extracted .ppn file lives in temp directory — could be cleaned by OS.
  - No error callback for user notification (just debugPrint).

─── 3F. THEME SYSTEM ──────────────────────────────────────────────────────────

  WHAT'S GOOD:
  • Full design-tokens file with semantic naming (AppColors, AppRadius, etc.).
  • Dynamic seed color with ColorScheme.fromSeed (Material 3 compliant).
  • Light and dark themes fully specified (AppBar, Card, Input, Button styles).
  • Google Fonts (Manrope) applied globally.
  • Theme mode persisted to SharedPreferences.
  • Page transitions configured per platform (Zoom on Android/Windows, 
    Cupertino on iOS).

  WHAT'S BAD:
  • PageTransitionsTheme is defined THREE times: once in ThemeProvider's 
    lightTheme, once in darkTheme, AND once more in main.dart's copyWith().
    The main.dart copies OVERWRITE the ThemeProvider versions. This is 
    redundant — define it only in ThemeProvider.
  • Some screens use hardcoded colors instead of Theme.of(context) — 
    e.g., Color(0xFF...) instead of colorScheme.surface.
  • AppDurations constants exist but most screens use inline Duration() 
    values instead of the tokens.
  • withOpacity() used in dark theme which is deprecated behavior — use 
    Color.withValues() or predefined alpha colors.

─── 3G. PERFORMANCE ───────────────────────────────────────────────────────────

  ISSUES FOUND:

  1. MISSING CONST CONSTRUCTORS
     Most screens and widgets are missing `const` qualifier on constructors.
     This prevents the framework from short-circuiting widget rebuilds.
     Impact: Minor but adds up across 13 screens with deep widget trees.

  2. UNNECESSARY REBUILDS
     • ConsultantScreen rebuilds the entire message list on every streamed 
       token (setState on _messages). Should use a list key or builder.
     • HomeScreen Consumer<ConnectionService> triggers rebuilds even when 
       connection status hasn't changed (notifications channel).

  3. NO PAGINATION
     All data queries fetch everything at once:
     - Sessions: all sessions loaded upfront.
     - Entities: all entities + attributes + relations fetched at once.
     - Consultant history: all logs fetched per session.
     For now this is fine (single user, limited data), but will degrade as 
     data grows.

  4. IMAGE HANDLING
     ProfileCompletionScreen uploads images without compression. A 10MB 
     phone photo goes straight to Supabase Storage. Should resize/compress 
     before upload.

  5. REALTIME OVERHEAD
     HomeScreen and NewSessionScreen can have concurrent Realtime listeners.
     These are WebSocket connections — mobile devices should minimize open 
     socket count.

─── 3H. ACCESSIBILITY ─────────────────────────────────────────────────────────

  WHAT'S PRESENT:
  • Good contrast ratios in both light and dark mode.
  • Touch targets are generally 48px+ (Material guidelines).
  • Semantic color usage (success/error/warning).

  WHAT'S MISSING:
  ✗ No Semantics widgets anywhere — screen readers will struggle.
  ✗ No tooltip text on icon buttons (except MicToggleButton).
  ✗ Status indicators are color-only (red/green dots) — need text labels.
  ✗ No alt text on images (logos, avatars).
  ✗ No keyboard navigation support (all gesture-based).
  ✗ No semantic headings structure for TalkBack/VoiceOver.

  This is acceptable for an FYP but would be required for production.

─── 4A. ARCHITECTURE & SERVICES ───────────────────────────────────────────────

  File: server/new_server.py (~1800 lines, active)
  File: server/server.py (~700 lines, legacy/baseline)

  FIVE SERVICE CLASSES (all in new_server.py):

  1. GraphService
     • Manages per-user NetworkX directed graphs.
     • Loads/saves graph as JSONB to knowledge_graphs table.
     • Semantic node search using SentenceTransformer embeddings 
       (cosine similarity, threshold 0.3) — big upgrade from server.py 
       which only did substring matching.
     • find_context() returns related facts for a query.
     • update_local_graph() merges new entities/relations into the graph.

  2. VectorService
     • Encodes text with all-MiniLM-L6-v2 (384-dim embeddings).
     • Stores to memory table with pgvector.
     • search_memory() via match_memory RPC (cosine similarity).
     • Uses asyncio.to_thread() for CPU-bound encoding — good.

  3. BrainService
     • Groq client with dual-model strategy:
       - llama-3.1-8b-instant: extraction, advice, voice commands (~100ms).
       - llama-3.3-70b-versatile: consultant queries (~3s).
     • JSON mode (response_format={"type":"json_object"}) for all extraction.
     • Methods: get_advice(), ask_consultant(), extract_entities_full(), 
       extract_events(), generate_summary(), detect_conflicts().
     • Temperature calibration: 0.1 (extraction), 0.6 (advice), 0.7 (consultant).

  4. SessionService
     • Creates sessions (live_wingman / consultant modes).
     • Logs individual turns to session_logs with role/speaker/confidence.
     • Rolling summarization every 20 turns (overwrites session summary).
     • Fetches consultant history for context injection.
     • Bulk message logging for save_session endpoint.

  5. EntityService
     • Persists structured entities to SQL tables.
     • Fuzzy deduplication via SequenceMatcher (0.85 ratio threshold).
     • Upserts entity attributes (key-value pairs, one value per key).
     • Creates directed entity relations.
     • Saves conflicts to highlights table.
     • Saves extracted events to events table.

  WHAT'S GOOD:
  • Service-oriented design with clear separation of concerns.
  • Each service owns its own DB connection logic.
  • Async-ready: asyncio.gather() for parallel graph+vector context fetch.
  • Dual-model strategy is cost-effective and latency-aware.

  WHAT'S BAD:
  • All five services + all eleven endpoints + all models + Ngrok setup + 
    LiveKit worker are in a SINGLE 1800-line file. This is the definition 
    of a God File.
  • Global mutable state: LIVE_SESSIONS dict and TURN_COUNTERS dict are 
    module-level globals with no cleanup mechanism — grow unbounded.
  • The LiveKit Agent Worker (entrypoint() function, ~100 lines) appears 
    to be dead code — comments say "might be unused if we switch to 
    client-side Deepgram". It is not wired up and should be removed.
  • nest-asyncio dependency exists only to patch event loop conflicts — 
    this shouldn't be needed with proper async design.

─── 4C. LLM INTEGRATION ──────────────────────────────────────────────────────

  WHAT'S GOOD:
  • System prompts are well-crafted: 
    - Explicitly tell LLM to hide implementation details ("don't mention 
      vectors, graphs, or embeddings").
    - Inject full context: graph facts + vector memories + session history 
      + past summaries.
    - Clear behavioral instructions ("2-4 sentences, natural language").
  • JSON mode prevents hallucinated free-text in extraction endpoints.
  • Temperature tuned per use case (0.1 extraction, 0.6 advice, 0.7 deep).
  • Token limits set per endpoint (60 for advice, 400 for consultant, 
    800 for extraction).

  WHAT'S BAD:
  • No retry logic: If Groq rate-limits or has a transient error, the 
    request just fails. No exponential backoff.
  • No token counting: Context could silently exceed model token limits.
    With 10 graph facts + 5 history messages + 3 summaries + system 
    prompt, it's easy to exceed 8K tokens, especially for llama-3.1-8b.
  • Max tokens are hardcoded — 400 for consultant may be too strict for 
    detailed explanations.
  • No hallucination verification: extract_entities_full() trusts the LLM 
    output completely. A hallucinated entity gets persisted to the DB.
  • Fallback string "WAITING" returned on Brain errors — the client 
    cannot distinguish between "no advice yet" and "server error".
  • No response caching — identical questions hit Groq every time.

─── 4D. DATABASE INTERACTIONS ─────────────────────────────────────────────────

  WHAT'S GOOD:
  • Uses Supabase Python SDK (parameterized queries via ORM methods).
  • Service key used server-side (bypasses RLS, correct for backend).
  • Proper upsert logic for entities (on_conflict with canonical_name).
  • match_memory RPC leverages pgvector cosine similarity natively.
  • Session-aware logging with proper foreign keys.

  WHAT'S BAD:
  • No transactions: Entity + attributes + relations are 3 separate 
    upserts with no rollback. If step 2 fails, you get an entity with 
    no attributes — partial state corruption.
  • Client-side JOINs: EntityScreen fetches entities, then attributes, 
    then relations in 3 separate queries. Should use SQL JOINs or a 
    single Supabase select with nested references.
  • fetch_session_summaries() fetches 3 sessions with their summaries — 
    no pagination, no cursor-based fetching.
  • No soft deletes anywhere. Hard deletes cascade, which is destructive.
  • No database indexing strategy documented. The entities table has a 
    unique constraint on (user_id, canonical_name), but there are no 
    explicit indexes on session_logs.session_id, memory.user_id, etc.
    PostgreSQL auto-creates indexes on foreign keys in some cases but 
    not all.
  • Rolling summarization overwrites the session's summary field — 
    previous summary versions are lost forever.

─── 4E. STREAMING & WEBSOCKETS ────────────────────────────────────────────────

  IMPLEMENTATION: SSE via FastAPI StreamingResponse for consultant streaming.

  PATTERN (COMPLEX):
  1. Python thread runs sync Groq streaming.
  2. Thread pushes tokens into asyncio.Queue via run_coroutine_threadsafe.
  3. Async generator awaits queue.get() and yields SSE lines.
  4. After stream ends, post-processing runs (log, vector save, graph update).

  WHAT'S GOOD:
  • Tokens stream to client one-by-one (smooth typing effect).
  • Final "done" event carries session_id back to client.
  • Post-processing (save memory, update graph) runs after stream completes.

  WHAT'S BAD:
  • Threading + asyncio mixing is overly complex and fragile.
  • If client disconnects mid-stream, the thread keeps running (no 
    cancellation mechanism).
  • No SSE heartbeat — connection could timeout during long silences.
  • No client backpressure — tokens queue up unbounded if client is slow.
  • Daemon thread means abrupt termination if server shuts down.
  • Error in the sync thread is not propagated to the SSE response.

  BETTER APPROACH: Use the async Groq client (if available) or 
  asyncio.to_thread() with a simpler generator pattern.

─── 4F. ERROR HANDLING ────────────────────────────────────────────────────────

  CURRENT PATTERN: Try-except-continue everywhere. Errors are logged with 
  emoji prefixes and the function returns a safe default value.

  WHAT'S BAD:
  • No distinction between transient errors (retry) and permanent errors 
    (fail fast).
  • No structured error responses to clients — just default strings.
  • Console logging with emojis is not machine-parseable or monitorable.
  • No error tracking (Sentry, etc.) — bugs in production are invisible.
  • Silent failures: client sees "WAITING" and has no idea if it's a 
    network error, a Groq API issue, or a database problem.
  • Rate limiting is completely absent — an attacker can spam /ask_consultant 
    and exhaust the Groq API quota (real money cost).

    VoiceAssistantService matches voice commands with contains("wingman"),

    wingman" would start a session.

  ✗ NO AUTHENTICATION ON SERVER ENDPOINTS
    All 11 endpoints accept a user_id parameter with zero verification.
    Anyone who knows the server URL can call /ask_consultant with any

  ✗ NO RATE LIMITING
    Every endpoint can be called unlimited times. An attacker can exhaust 
    the Groq API quota (real money) by spamming /ask_consultant.

    call the API endpoints. Old server.py was actually better here — it 
    read from ALLOWED_ORIGINS env var.

    Server returns "WAITING" for errors. Client shows SnackBars sometimes, 
    dialogs other times, or silently continues. No unified error protocol 
    between frontend and backend.

    All list fetches (sessions, entities, consultant history) load 
    everything at once. Will degrade with data growth.

  ✗ AUDIO ENCODING SUBOPTIMAL

    reduce transcription quality.

  ✗ HARDCODED VALUES EVERYWHERE
    • Magic numbers: 30s timeout, 4s auth delay, 300 velocity, 0.85 fuzzy 
      threshold, 0.3 cosine threshold, 20-turn summary interval.
    • Hardcoded strings: route names, "Bubbles", "Live Wingman", etc.
    • Hardcoded lists: 40+ countries in profile_completion_screen.
    • None of these are in constants files or configurable.

 8. THE UGLY (CRITICAL ISSUES)
════════════════════════════════════════════════════════════════════════════════

  🔴 .ENV FILE BUNDLED AS FLUTTER ASSET
     The pubspec.yaml includes ".env" in the flutter assets list. This 
     means the .env file (containing Supabase keys, Deepgram API key, 
     and Picovoice access key) is compiled INTO the APK/IPA. Anyone who 
     decompiles the app can extract all API keys. This is extremely 
     dangerous even for a development build.

  🔴 SERVER KEYS HARDCODED AS FALLBACK DEFAULTS
     new_server.py has real API keys as default values in os.getenv() calls:
       DEEPGRAM_KEY = os.getenv("DEEPGRAM_KEY", "8f4f1c36dd57...")
       GROQ_KEY     = os.getenv("GROQ_API_KEY", "gsk_0V8yw5KK...")
       LIVEKIT_SECRET = os.getenv("...", "jL8bwZ2fqLxC...")
       SUPABASE_KEY = os.getenv("...", "eyJhbGciOiJI...")
     These keys are in the Git repository. Even though this is intentional 
     for development, these keys work with live services and anyone with 
     repo access has full API control.

  🔴 NO INPUT SANITIZATION ON LLM PROMPTS
     User input (questions, transcripts) goes directly into LLM system 
     prompts without sanitization. Prompt injection attacks could instruct 
     the model to ignore system instructions, return fabricated data, or 
     manipulate the knowledge graph.

  🔴 THREAD-ASYNC MIXING IN STREAMING
     The /ask_consultant_stream endpoint uses threading.Thread + 
     asyncio.Queue + run_coroutine_threadsafe. This is a recipe for 
     deadlocks, race conditions, and resource leaks. The daemon thread 
     continues running even if the client disconnects.

  🔴 GLOBAL STATE WITHOUT CLEANUP
     LIVE_SESSIONS and TURN_COUNTERS are module-level dictionaries that 
     grow indefinitely. No TTL, no cleanup, no max-size. Over time, these 
     will consume increasing memory until the server is restarted.

  🔴 NO TRANSACTION SAFETY
     Entity creation involves 3+ separate database calls (upsert entity, 
     upsert attributes, upsert relations). If any call fails, the data is 
     left in a partially saved state. There is no rollback mechanism.

  🔴 VERIFY EMAIL CALLS SIGNUP AGAIN
     verify_email_screen.dart "resends" verification by calling 
     signUpWithEmail() with the stored email and empty password. This is 
     unpredictable — could create session conflicts or authentication 
     errors depending on Supabase state.

 12. FINAL VERDICT
════════════════════════════════════════════════════════════════════════════════

  OVERALL GRADE: B+

  This is an ambitious, technically impressive project that successfully 
  integrates cutting-edge AI infrastructure (Groq, LiveKit, pgvector, 
  Porcupine) into a polished mobile application. The dual-mode Wingman/
  Consultant design is architecturally sound, the knowledge graph + 
  vector memory combination is genuinely novel for a mobile app, and 
  the UI/UX quality is well above average for an FYP.

  The main weaknesses are engineering discipline issues rather than 
  design flaws: missing tests, business logic in UI, hardcoded values, 
  and the monolithic server. These are typical of a fast-moving student 
  project and are all fixable without architectural changes.

  WHERE IT SHINES:
  • The "brain" architecture (graph + vectors + dual LLM) is the standout.
  • The voice pipeline (wake word → STT → LLM → TTS) works end-to-end.
  • The Flutter UI is genuinely polished with thoughtful animations.
  • Provider architecture is competent and avoids common pitfalls.

  WHERE IT NEEDS WORK:
  • Security (.env in assets, no server auth, CORS wildcard).
  • Testing (zero coverage on both sides).
  • Server engineering (monolithic, no rate limits, fragile streaming).
  • Code organization (magic numbers, dead code, business logic in UI).

  The gap between "impressive FYP demo" and "production-ready app" is 
  primarily about engineering hygiene, not functionality. The feature set 
  is complete and the architecture can scale — it just needs hardening.
